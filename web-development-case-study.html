Learning from a real refactor

Attack Momentum frontend case study

This document explains what existed before, why some choices were limiting the project, what was improved, and what engineering lessons you can take from the session.

1. Project Context

Attack Momentum is a static frontend that shows live football matches, SofaScore attack momentum widgets, scores, and statistics. A separate Fastify proxy requests SofaScore data because the browser cannot safely and reliably call the upstream API directly.

Frontend

A vanilla HTML, CSS, and JavaScript app hosted as static files, currently suitable for GitHub Pages. It renders match cards, filters by league, loads widgets, and refreshes live scores and stats.

Backend Proxy

A Fastify server that exposes endpoints like /api/live and /api/:eventId/stats. It hides SofaScore scraping details from the frontend and can enforce cache, CORS, rate limits, and request policy.

Senior frontend work is not only about making things prettier. It is about aligning rendering, data fetching, user interaction, performance, and security boundaries with the real purpose of the product.

2. What Was There Before

The previous app was already useful: it showed live matches, embedded attack momentum graphs, displayed selected stats, supported a league dropdown, and had basic drag-and-drop card organization. The architecture was intentionally simple: one HTML file, one CSS file, and a few JavaScript modules.

Original Interaction Model

  • Load live events from the proxy.
  • Create one card for each eligible match.
  • Put a SofaScore iframe directly in each card.
  • Refresh scores and stats on timers.
  • Filter by league by changing card display styles.

Original Code Shape

  • DOM was built imperatively in scripts/app.js.
  • API calls used Axios loaded from a CDN.
  • Drag-and-drop swapped innerHTML between cards.
  • Filtering reloaded iframes by resetting their src.
  • The proxy URL was editable from the UI after the first refactor pass.

None of those choices are unusual for an early prototype. They are fast to write, easy to understand at first, and good enough to prove that the idea works. The problem is that prototype choices often become production bottlenecks if they are not revisited.

3. What Was Wrong With It

Performance problems

  • All graph iframes loaded immediately. Iframes are expensive because each one creates a separate embedded browsing context. Loading many of them on page start makes the app slower even before the user scrolls.
  • Filtering could reload iframes. Resetting an iframe src when a card becomes visible costs network, CPU, and time. A filter should usually change visibility, not reload content.
  • Stats requests had weak flow control. Running many async requests from a loop can overwhelm the proxy and make the UI harder to reason about when many matches are live.

Maintainability problems

  • Drag-and-drop swapped HTML strings. This breaks the mental model of the DOM. Event state, selected periods, iframe state, and future component state can be lost or duplicated.
  • The page had no clear state model. It relied heavily on querying and mutating scattered DOM nodes. That makes future changes riskier.
  • UI and data behavior were mixed together. For example, filtering also touched iframe loading. Good UI code tries to keep unrelated effects separate.

Usability problems

  • The user had to visually scan many cards without a search box.
  • The status of refreshes was mostly hidden in the console.
  • The old layout asked users to rely on browser zoom instead of responsive design.
  • The stats were split by home and away columns, which made comparison slower.

Security and exposure problems

  • The proxy initially reflected any browser Origin in CORS headers, so any site could make browser requests to the proxy if it knew the URL.
  • The temporary visible API URL control was useful during development, but it exposed too much implementation detail to users.

Important architectural lesson: hiding a URL in frontend code is not real security. If the browser can request it, users can inspect it. Protection must live on the server through origin policy, rate limits, auth, quotas, caching, and abuse detection.

4. How The Improvements Were Implemented

  1. Reworked the page structure

    The app header became simpler, and the main controls became a dashboard toolbar: league filter, search field, refresh button, match count, status text, and last update time.

  2. Moved from CDN Axios to native fetch

    The app no longer needs a third-party script just to make HTTP requests. Modern browsers already provide fetch, AbortController, and JSON parsing. 

    const controller = new AbortController();
const timeout = window.setTimeout(() => controller.abort(), 15000);

const response = await fetch(`${baseUrl}${path}`, {
    headers,
    signal: controller.signal
});

  3. Introduced state-driven rendering

    Instead of rebuilding or mutating cards blindly, the app tracks live matches, reuses existing cards when possible, removes stale cards, and updates only the fields that changed.

  4. Lazy-loaded graph iframes

    The app now creates iframes with a data-src first, then hydrates the real src only when the card gets close to the viewport. 

    const observer = new IntersectionObserver((entries) => {
    entries.forEach((entry) => {
        if (!entry.isIntersecting) return;
        hydrateIframe(entry.target);
        observer.unobserve(entry.target);
    });
});

  5. Limited parallel stats requests

    Stats refreshes now run with a small worker pool. This protects the proxy from bursts while still updating multiple visible cards efficiently.

  6. Replaced HTML-swapping drag-and-drop

    Drag-and-drop now moves actual card nodes in the DOM using insertBefore. This keeps each card's iframe, selected period, event listeners, and state intact.

  7. Improved stats presentation

    Stats became one comparison table per card: home value, stat label, away value. The leading value gets highlighted, making scanning faster.

  8. Removed the visible API control

    The public page no longer shows or accepts a proxy URL override. That keeps the interface focused and avoids teaching users to interact with internal infrastructure.

  9. Tightened proxy CORS behavior

    The Fastify proxy now rejects /api/* browser requests that do not come from an allowed origin. GitHub Pages is allowed, and localhost origins remain allowed for development. 

    const isApiRequest = request.url.startsWith("/api/");
const isAllowedOrigin = origin && allowedOrigins.includes(origin);

if (isApiRequest && !isAllowedOrigin) {
    return reply.status(403).send({ error: "Forbidden origin" });
}

5. Results Brought By The Changes

Less initial work Iframes are loaded near the viewport instead of all at once.
More stable UI Filtering no longer reloads graph widgets unnecessarily.
Safer interactions Drag-and-drop preserves card nodes instead of replacing HTML strings.
Better scanning Search, status, match counts, and highlighted stat leaders reduce cognitive load.

Validation performed

  • JavaScript syntax checks passed with node --check.
  • The app loaded in Chromium with mocked API responses and no console/page errors.
  • Search, filtering, stat rendering, empty state, and responsive layout were checked.
  • The proxy returned 403 for disallowed or missing origins and allowed the GitHub Pages preflight.

The result is still a vanilla static app, but it behaves more like a real monitoring tool: controls are visible, updates are understandable, and expensive work is deferred or controlled.

6. Security Lesson: Frontend Visibility vs Backend Enforcement

A common misconception is that removing a visible button or hiding a URL from the UI secures an API. It improves presentation, but it does not create a security boundary.

Not Real Security

  • Removing a button.
  • Minifying JavaScript.
  • Using an obscure ngrok URL.
  • Relying on users not opening DevTools.

Better Server-Side Controls

  • Strict CORS for browser clients.
  • Rate limiting by IP or token.
  • Short cache TTLs to reduce upstream load.
  • Request logging and abuse detection.
  • Optional authentication for private usage.

CORS blocks normal browser-based abuse from other websites, but it does not stop scripts, curl, bots, or custom clients from sending forged headers. Treat CORS as one layer, not the whole defense.

7. Web Development Lessons For The AI Era

Understand the system shape Before changing code, map the architecture: static frontend, proxy server, upstream API, browser widgets, and deployment.
Separate concerns Filtering should filter. It should not reload iframes. Dragging should reorder nodes. It should not rewrite card HTML.
Make costs visible Iframes, network requests, timers, and DOM mutations all have costs. Good architecture controls when those costs happen.
Design for the real workflow This app is not a marketing page. It is a live match monitor, so density, scanning, filtering, and status matter most.
Use AI as leverage, not autopilot Ask AI to explain tradeoffs, but verify behavior with tests, browser checks, and your own understanding.
Protect at the right layer Frontend code improves UX. Backend code enforces policy. Do not confuse hiding controls with securing infrastructure.

Questions to ask during any refactor

  • What user workflow is this interface really optimizing for?
  • Which operations are expensive, and when do they happen?
  • What state does the UI need to preserve?
  • What belongs in frontend code, and what must be enforced on the server?
  • How will I verify that the change works?
  • What did I make easier for the next developer?

8. Good Next Improvements

The current refactor improves the app without introducing a framework. That was the right scope for a small static project. If the project continues growing, these are strong next steps:

  • Add proxy-side rate limiting, especially for public ngrok exposure.
  • Move the API base URL to deploy-time configuration instead of editing source for each tunnel.
  • Add lightweight frontend tests for filtering, period switching, and card rendering.
  • Persist user card order and dismissed matches in local storage.
  • Add a "show hidden matches" action so dismissing a card is reversible.
  • Consider a small framework only if state and UI complexity continue growing.

The senior-level skill is not "always use React" or "never use frameworks." The skill is recognizing the current complexity level and choosing the smallest architecture that handles it well.
